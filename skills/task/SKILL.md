# task Skill - タスク定義の作成

## 変数定義
- `$D` = `.claude/claude-dev-orchestrator` — 生成物のベースディレクトリ

## 概要
ユーザーの要望をヒアリングし、`$D/tasks/` ディレクトリにタスク定義ファイルを作成する。

## 入力
- `$ARGUMENTS`: タスクの概要（自然言語）

## 手順

### 1. コンテキスト収集
- `$D/tasks/` ディレクトリの既存タスクファイルを確認し、ID・形式を把握する（なければ新規作成）
- CLAUDE.md を読み、プロジェクトの規約・技術スタックを把握する
- `.claude/dev-orchestrator.yml` があれば読み、プロジェクト設定を把握する

### 2. 対象コードの調査
実装に必要な情報を網羅的に調査する。以下の観点を重点的に確認する:
- **変更対象ファイル**: パス・関数名・行番号を特定する
- **import/依存関係**: 変更対象が依存するモジュール・変更対象に依存するモジュールの両方を追跡する
- **類似機能の既存実装**: 参考にすべきパターンやコード規約を把握する
- **環境変数・設定ファイル**: 現状の値と変更が必要かを確認する
- **既存テストの有無と場所**: テストファイルのパスと関連テストケースを特定する
- 調査結果をもとに要件・受け入れ条件を具体化する（ファイルパスや関数名を正確に記載する）

### 3. タスク定義の作成
次のIDは `$D/tasks/task-*.md` の最大番号 +1 とする。

以下の形式でタスク定義ファイルを組み立てる:

```markdown
---
status: pending
---
# task-N: 短い名前（kebab-case）
タスクの概要を1-2文で説明。

## 要件
- 具体的な実装要件をリストで記載
- ファイルパスや関数名など、可能な限り具体的に
- Agent が迷わず着手できるレベルの粒度にする

## 受け入れ条件
- テストが通ること
- 具体的な動作確認項目
```

### 4. ユーザーに確認
- 作成したタスク定義を表示してユーザーに確認を求める
- フィードバックがあれば修正する

### 5. タスクファイルを作成
- ユーザーの承認後、`$D/tasks/task-N.md` として保存する

## タスク定義のガイドライン
- **1タスク = 1PR** になる粒度にする。大きすぎる場合は分割を提案する
- 要件は Agent が自律的に実装できる具体性を持たせる（「適切に実装」のような曖昧な表現は避ける）
- 受け入れ条件は機械的に検証可能なものを優先する
- 既存コードのパスや関数名は調査結果をもとに正確に記載する
